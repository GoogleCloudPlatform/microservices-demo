# Implementation Plan: Recently Viewed Strip on Product Detail Page

**Branch**: `004-recently-viewed-strip` | **Date**: 2026-05-20 | **Spec**: [spec.md](spec.md)

## Summary

Surface a "Recently Viewed" strip on the product detail page showing up to 5 products the shopper has visited earlier in the current session, so they can return to an item without retracing their steps. The list of viewed product IDs is stored in a new browser cookie (`shop_recently_viewed`) — consistent with the existing currency and session-ID cookies — and product details are resolved via the existing `getProduct()` RPC. The change is contained entirely within the frontend service: one updated handler, two new template files, and minor CSS additions.

## Technical Context

**Language/Version**: Go 1.21 (frontend service)
**Primary Dependencies**: Standard library only — `net/http`, `strings`; existing `*pb.Product` protobuf type; existing `getProduct()` RPC already in `rpc.go`
**Storage**: Browser cookie `shop_recently_viewed` (pipe-separated product IDs, max-age 48 h) — no server-side storage
**Testing**: Go `testing` package + existing `go test ./...` invocation; manual browser walkthrough via `skaffold dev`
**Target Platform**: Existing GKE deployment, unchanged
**Project Type**: Web service (frontend Go server, server-rendered HTML templates)
**Performance Goals**: Strip render adds at most 5 serial `getProduct()` gRPC calls — identical to the existing recommendations flow; no new latency budget required
**Constraints**: No new services, datastores, or infra changes (HC-001, HC-002, HC-003)
**Scale/Scope**: Single frontend service change; no cross-service coordination

## Phase 1 — Design (complete)

See [data-model.md](data-model.md) and [quickstart.md](quickstart.md).

**No external contracts to define.** The feature is fully internal to the frontend service — no new gRPC methods, REST endpoints, or protobuf changes.

### Key design decisions

1. **Cookie write happens before template render** — the cookie is updated (with the current product prepended) before the strip list is assembled for the template, so the strip always excludes the current product (`updatedIDs[1:]`).

2. **Graceful degradation** — `getRecentlyViewedProducts` returns whatever it has fetched if a single `getProduct()` call fails, matching the non-critical error handling already used for recommendations (`log.Warn`, no HTTP 500).

3. **No JavaScript** — the feature is entirely server-rendered, consistent with the rest of the frontend.

4. **CSS mirrors `.recommendations`** — the new `.recently-viewed` class reuses the same grid layout, keeping visual consistency without introducing new design tokens.
